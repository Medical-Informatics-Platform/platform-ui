# Conventions

## Confirmed Conventions

### Angular and TypeScript
- The app uses Angular 21 standalone components.
- `app.config.ts` provides router, HTTP client, XSRF, zoneless change detection, and ECharts.
- TypeScript is strict. `tsconfig.json` enables strict templates, no unused locals/parameters, no implicit returns, and related checks.
- Component style language is CSS.
- Tests use Jasmine/Karma and Angular `TestBed`.

### State and Services
- Signals are the dominant state primitive for feature state.
- Services use `inject()` heavily for dependencies.
- `ExperimentStudioService` owns experiment composition state and persistence to sessionStorage.
- `ExperimentsDashboardService` owns paged dashboard state and experiment list mutations.
- `ErrorService` is used for global feature errors.

### Routing and Access
- Routes are lazy-loaded with `loadComponent`.
- Route guards enforce auth, terms/NDA acceptance, and selected onboarding behavior.
- Optional notebook access is gated both in Angular routing and nginx runtime config.

### API and Runtime Config
- API calls use same-origin relative `/services/...` URLs.
- The credentials interceptor adds `withCredentials` for same-origin API calls.
- Runtime env is read from `window.__env`, generated by `docker-entrypoint.sh` in containers.
- `RuntimeEnvService` is the preferred abstraction for app code that needs runtime env values.

### UI and Styling
- Use component CSS for feature-specific styles.
- Use `src/styles.css` for app-wide styles and QueryBuilder theming.
- Consult `DESIGN_SYSTEM.yaml` before visual changes.
- Existing branding assets live under `src/assets`.

### Visualization and Algorithm Results
- Backend algorithm definitions are mapped in `src/app/core/algorithm-mappers.ts`.
- Result table and chart rendering use registry patterns under `src/app/pages/experiment-studio/visualisations`.
- Legacy algorithm keys may exist for historical stored experiment compatibility.

### Testing Patterns
- Specs are colocated with source files.
- Component tests commonly use `TestBed`, service stubs, and Signals.
- Add focused regression tests for mapper, registry, and service behavior changes.

## Recommended Conventions To Verify
- Add dedicated `lint`, `format`, and `typecheck` scripts if the team wants separate quality gates.
- Add a pull-request CI workflow that runs at least `npm ci`, `npm run build`, and `npm test`.
- Establish shared fixture builders for repeated backend experiment/result payloads.
- Prefer typed interfaces over `any` in new code where existing backend payload uncertainty does not require flexibility.
- Add e2e coverage for login-protected flows if a stable test identity/backend environment is available.
